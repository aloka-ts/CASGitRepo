This package contains the Event classes representing the component primitives and their specified parameters, if the coding of a paramater is not included in any specific class, it is common to all the specified components and therefore coded within that components parent class. The parent classes can be either ComponentIndEvent or ComponentReqEvent depending on whether the component is an indication or request primitive where:
  • Indication component primitives - are primitives that can only be sent from a JainTcapProvider. to a JainTcapListener
  • Request component primitives - are primitives that can only be sent from a JainTcapListener to a JainTcapProvider.
A generic set of Components primitives has been created that abstracts the functionality of the Component primitives for the supported variants. The mapping between these generic primitives and the variant specific primitives they represent is illustrated in Table 1:

NOTE:

  • The InvokeReqEvent and the InvokeIndEvent can represent either a 'Last' or 'Not Last' Invoke Component primitive for the an ANSI implementation of the JAIN TCAP API, where a flag is included within the primitive to indicate the components function, this flag should be ignored within an ITU implementation.
  • Similarly, the ResultReqEvent and the ResultIndEvent can represent either a 'Last' or 'Not Last' Result Component primitive, therefore a flag is also included in these Events to indicate which case applies for both an ITU and ANSI implementation of the JAIN TCAP API.
  • Within the RejectIndEvent a flag will indicate whether the primitive is a 'User', a 'Local' or a 'Remote' Reject for an ITU implementation of the JAIN TCAP API.

Mapping of JAIN Mandatory and Optional parameters of Component primitives

The following tables identify the mapping between the ITU and ANSI TCAP Component primitives and parameters. The standards supported in JAIN TCAP v1.0 include:

  • ITU-T (03/93) Q series (Q.771-775) û SS7 Functional description of Transaction Capabilities.
  • ITU-T (06/97) Q series (Q.771-775) û SS7 Functional description of Transaction Capabilities.
  • ANSI (1992) T1.114.1 - T1.114.5 û SS7 Transaction Capabilities Application Part.
  • ANSI (1996) T1.114.1 - T1.114.5 û SS7 Transaction Capabilities Application Part.

Mandatory and Optional Parameters

Each of the supported standards defines mandatory and optional parameters for each primitive. As JAIN TCAP supports all the mandatory and optional parameters of each primitive of the supported variants within the one specification, it is necessary to redefine the mandatory and optional parameters of the standards. This new definition of mandatory and optional parameters will be referred to as JAIN TCAP mandatory and optional and will accommodate the mandatory and optional parameter differences within the standards supported.

JAIN TCAP MANDATORY: A JAIN TCAP mandatory parameter is a parameter within a primitive that is mandatory for all the supported variants. In the tables below if any parameter is mandatory (M) for all the variants then it will be mandatory for that JAIN TCAP primitive.

JAIN TCAP OPTIONAL: A JAIN TCAP optional parameter is a parameter within a primitive that is either optional or not supported by any of the supported variants. In the tables below if a parameter is regarded as mandatory for a specific variant and optional or not supported by the other variant it will be regarded as a JAIN TCAP optional.
